作者： wwk (C++ R&D Engineer)

设备： Redmi Note 7 Pro (代号: violet)

系统环境： LineageOS 23.2 (Android 16) + Magisk Root

核心目标： 打造一台 7x24 小时运行、支持原生 Docker、拥有完整 C++ 编译环境的便携 WebServer 服务器。

写在前面： 之前写过一篇文章是免root的，虽然开始上手简单，但是不能运行docker，Termux模拟的半吊子linux也不能获取root,总而言之如隔靴搔痒，怎么不得劲怎么来。所以有了这篇文档记录了将一台退役的骁龙 675 手机彻底改造为 Linux 容器服务器的全过程。从底层的 Bootloader 到内核层的 Cgroups 注入，再到应用层的 Android 16 杀后台破解，一路踩坑无数。谨以此文献给所有热爱折腾、不甘心让旧设备在抽屉里吃灰的极客们。

阶段一：破除枷锁与夺取底层控制权

要把手机变成服务器，第一步就是打破厂商的牢笼，拿到最高权限。

1. 解锁 Bootloader 与刷入 Recovery

  • 解锁 BL：使用小米官方解锁工具，绑定账号并等待解锁周期（通常 168 小时）。这一步不难直接上网搜就行，总共就两步，但是后续一切操作的基石，还是要感谢雷军大帝，不过澎湃就要复杂些了，本文不展开讨论，也不建议这么折腾一遭，其实最好的选择就是把手机卖了，买个云服务器，但是折腾过来了就给同样还有ReadmiNote7Pro的同学们留下点东西吧。

  • 准备工作：

    • OrangeFox Recovery 镜像：专门为 Redmi Note 7 Pro (violet) 编译的版本（俗称“橙狐”）。强烈建议解压下载好的 ZIP 包，提取出里面的 recovery.img 文件放到电脑桌面备用。

2. 破釜沉舟：彻底清理旧战场 (格式化与传包)

为了后续能够顺利刷入极高版本的 LineageOS 23.2，并解决 Android 底层的数据加密冲突，这一步是无法避免的阵痛。

  • 格式化 Data (Format Data)：点击橙狐界面底部的垃圾桶图标 (Wipe)，选择 Format Data 选项卡，输入 yes 并确认。

    ⚠️ 破釜沉舟警告：执行 yes 的那一刻，手机上的所有照片、通讯录、旧系统数据将被瞬间化为乌有，相当于硬盘的低级格式化。

  • 重启到 Recovery：格式化完成后，千万不要立刻重启系统！依次点击 Menu -> Reboot -> Recovery，让橙狐在一个绝对干净的底层环境下重新启动。
  • 💡 极其关键的传包操作：此时手机内部存储已被完全清空。请用数据线连接电脑，电脑会将其识别为 MTP 设备。趁现在，将提前下载好的 LineageOS 安装包 和 Magisk 压缩包 复制到手机根目录。

3. 注入新魂：刷入 LineageOS 23.2 与 Magisk

  • 刷入系统与 Root：在橙狐中点击 Install，先选中 LineageOS 的 ZIP 包刷入。为了防止重启丢失权限，紧接着选中 Magisk 压缩包一并刷入。
  • 迎接新生：全部刷入完成后，点击 Reboot System，耐心等待第一次开机。彻底抛弃臃肿的 MIUI，一个极简的底层环境诞生了。

  • 部署 ACC (Advanced Charging Controller)：
    • 在 Magisk 中安装 ACC 模块。
    • 配置供电策略，开启 “电池直供” (Battery Idle Mode) 功能。
    • 原理：当插着充电线时，主板直接从电源适配器取电，绕过电池。彻底消除电池寿命焦虑和安全隐患，让它真正具备“服务器”的物理素质。

Docker 大法小试牛刀

0. 引言：为什么 C++ 开发需要 Docker？

在日常的 C++ 研发中，环境配置往往让人头疼。特别是在进行网络编程（如开发 WebServer）时，代码在 Windows 宿主机、Linux 虚拟机乃至最终的云服务器上，常常面临编译器版本不一致、依赖库缺失的问题。

引入 Docker 的核心目的，就是为了构建一个标准、隔离且用完即毁的“实验沙盒”。充分利用宿主机（如 i5-12600KF + 32GB RAM）的性能溢出，我们可以在不污染虚拟机系统的前提下，随意折腾不同版本的 GCC、CMake 和系统环境。

1. 踩坑记录：Docker 的权限管理

在 Ubuntu 虚拟机中安装完 Docker 后，新手最容易遇到的第一个拦路虎就是权限问题。

1.1 典型报错信息

当执行 docker run 时，终端抛出如下致命错误：

permission denied while trying to connect to the docker API at unix:///var/run/docker.sock

1.2 问题根源与解决思路

这是因为 Docker 的守护进程默认由 root 用户运行，其监听的 Unix 套接字文件 /var/run/docker.sock 只有 root 和 docker 用户组才有访问权限。

千万不要养成每次加 sudo docker 的坏习惯！ 这会导致容器内挂载生成的编译产物（如 .o 文件）全部变成 root 属主，后续在宿主机修改代码时会频频遇到 Permission Denied。

一劳永逸的解决方案：

执行完毕后，直接输入 docker ps，若无报错即代表提权成功。

2. 核心逻辑：区分镜像与容器

理解 Docker，必须分清两个概念：

  • 镜像 (Image)：静态的“只读模板”。例如 ubuntu:24.04，它相当于一个被压缩好的精简版操作系统根文件系统。
  • 容器 (Container)：镜像运行时的实体。它是动态的进程，拥有独立的文件系统、IP 地址和网络端口。

核心认知：容器是基于镜像生成的快照。一旦容器通过 docker run 创建并运行，它的挂载点等核心配置就已经锁死，无法动态向运行中的容器塞入新的挂载目录。

3. 核心命令速查表

日常高频使用的命令清单，建议熟记：

命令 作用 开发者场景
docker run -it [镜像] 创建并交互式启动新容器 获取一个纯净的编译环境
docker ps 查看正在运行的容器 检查 WebServer 容器是否挂掉
docker ps -a 查看所有容器（含已退出） 找回之前用过的历史环境
docker images 查看本地缓存的镜像 清理无用的旧版本镜像
docker start -i [名称] 唤醒并进入已停止的容器 昨天没写完的代码今天继续
docker stop [ID] 停止运行中的容器 释放占用的端口或内存
docker rm [ID] 删除已停止的容器 彻底销毁被玩坏的沙盒

5. 容器的优雅退出与重连

在容器内跑着服务端程序时，我们经常需要切回宿主机查看系统状态。

个人注意力恢复计划表

基于心理学极简主义的四个核心阶段，旨在通过减少认知摩擦来夺回注意力控制权。

阶段一：让认知噪音可见（审计注意力漏洞）

  • 行动：评估认知杂乱的第一步，是花一天时间记录注意力流失的瞬间。准备一张纸放在电脑桌旁，每次你下意识地拿起手机，或者在编译 C++ 代码、排查 WebServer 项目 Bug 的间隙习惯性地切换网页，就在纸上画个记号。
  • 目的：大脑倾向于简单和高效的编码以降低认知负荷。识别出那些“低价值、高频次”的微小决策和干扰源，可以帮助你从源头减轻认知负荷，恢复大脑的功能连贯性。

阶段二：减少摩擦（数字、物理与时间层面的极简）

将过去的习惯转化为稳定的时间惯例（即“时间默认值”），能让专注变得比分心更容易。

1. 早晨（时间层面的减少）

  • 07:55 - 08:30：通勤时间刚好 35 分钟。通过固定时间段来结构化每一天，能减少决策带来的认知消耗并防止注意力漂移。将这段时间设定为“英语专属模块”，只播放赖世雄英语课程。

2. 白天（数字层面的减少）

  • 行动：每一次通知或提示音都会迫使大脑评估是否需要转移注意力，这会带来极大的控制成本。工作期间，关闭所有非紧急 App 的通知，减少任务切换带来的“注意力残留”。
  • 10:30和16:30：统一处理微信与企业微信消息。

3. 晚间（物理层面的减少与睡眠防御）

  • 20:00 - 21:30：物理环境中的视觉杂乱会引发潜意识的微小决策，增加认知摩擦。把桌面上与编程无关的杂物清空，只保留 WebServer 项目所需的开发工具，以此让空间积极地支持你的专注。
  • 22:30 以后：针对影响睡眠的手机成瘾问题，执行最严格的物理隔离。卧室只作为休息区，把手机充电线放在客厅，坚决不把手机带上床，切断深夜无意识的屏幕浏览。

阶段三：固化习惯（谨慎的决策与获取）

  • 行动：在之前的减法生效后，要开始管理未来进入你生活的事物。在引入新的工具、承诺或购买物品前，运行一个简单的过滤器：
    • 它是必不可少的吗？
    • 它是耐用的吗？
    • 它会简化维护还是增加注意力负担？
    • (提示：在决定研究新的前沿组件，或者购买类似 Stadia 手柄的“大厂遗产”之前，先问问自己这些问题。)
  • 目的：这能防止注意力杂乱重新侵入系统。许多认知超载不仅来自我们已有的东西，更来自我们不断增加的新工具和新承诺。在前端拦截这些事务，可以提前保护注意力，避免后续产生维护成本。

阶段四：深化专注（建立元认知知识）

  • 行动：按照上海时间每周日晚上20:00，进行一次反思与重新校准。
  • 内容：
    • 记录本周哪些方法有帮助，哪些干扰又悄悄溜了回来。
    • 评估本周的通勤英语听力是否坚持得很好，晚上不带手机进卧室的规定执行了几天。
    • 规划一个可以在下一个周期调整的微小变动。
  • 目的：通过在低要求状态下定期的反思与训练，专注力会得到巩固并更容易被唤醒。专注将不再需要反复召唤，而是由一个要求更少的环境来自然支撑。

❓ 常见问题排查手册 (Troubleshooting)

  1. Nginx 报错 invalid PID number：
    • 原因：Nginx 进程没在运行。
    • 解决：直接输入 nginx 启动，不要用 reload。
  2. 外网访问博客转圈圈 (Timeout)：
    • 原因：路由器拦截了入站连接。
    • 解决：进入路由器后台，关闭 IPv6 防火墙，或添加允许 8888 和 21116 端口的规则。
  3. RustDesk 提示“Key 不匹配”：
    • 原因：服务端重新生成了密钥。
    • 解决：检查手机上 id_ed25519.pub 的内容是否变动，更新客户端设置。
  4. curl 本地通，外网不通：
    • 检查 Nginx 配置里是否写了 listen [::]:8888; (必须有 [::])。

告别向日葵、ToDesk卡顿：我用阿里云2M小水管自建了RustDesk远程桌面(Docker版)

抛弃 TeamViewer，我用阿里云 2M 小水管自建了 RustDesk 远程桌面

前言

作为一名 C++ 开发者，远程控制家里的电脑是刚需。受够了向日葵和todesk的商业限制和限速以及高峰时段的不稳定，我决定尝试开源的 RustDesk。

https://github.com/rustdesk/rustdesk 这是一个非常优秀且热门的开源远程控制项目，在 GitHub 上拥有极高的 Star 数。

如果只是下载即用（使用官方免费公共服务器），体验可能一般；但如果你愿意稍微折腾一下（自建服务器或配合虚拟组网），它的体验可以极其流畅，甚至超越 向日葵或 ToDesk。

本文记录了我在阿里云一台 2Mbps 带宽 的试用白嫖服务器:smirk:上部署 RustDesk Server 的全过程，最终实现了 1080p 画质下 90ms 左右 的稳定延迟。

1. 准备工作

  • 服务器：阿里云 ECS（CentOS 7/Ubuntu），位于国内且离你越近越好，目前可登录阿里云申请使用。
  • 配置：2核 2G，带宽 2Mbps（这是关键瓶颈）。
  • 本地环境：shandong（家里和公司），服务器是深圳。
  • 目标：在公司流畅控制家里的 Windows 电脑。

3. 实测体验与数据

延迟表现

  • 物理链路：济南 -> 深圳(服务器) -> 济南。
  • 平均延迟：93ms 左右。虽然走了几千公里的回头路，但这个延迟，处理文档、写代码完全没问题。

带宽压力测试 (2Mbps 够用吗？)

我的服务器带宽只有 2Mbps (峰值下载速度约 250KB/s)。

  • 静止画面：流量几乎为 0。
  • 日常办公：流量在 50KB/s - 100KB/s 波动，非常流畅。
  • 看视频/拖动窗口：1080p 画面剧烈变化时，流量瞬间飙升到 200KB/s，直接跑满带宽。
    • 结论：2Mbps 是流畅体验 1080p 的“及格线”。如果你的带宽只有 1Mbps，建议降低分辨率或画质。

4. 优化建议

  1. 画质设置：在 RustDesk 顶部菜单选择“平衡”或“最佳速度”，可以显著降低带宽占用。
  2. P2P 直连：如果两边网络环境允许（有 UPnP 或 IPv6），RustDesk 会尝试直连（绿色图标），那样延迟能降到 30ms 以内且不占服务器带宽。
  3. Docker 镜像：国内机器一定要用加速镜像（如 DaoCloud），否则 docker run 会卡到怀疑人生。

总结

RustDesk + 闲置云服务器是一个非常高性价比的方案。虽然 2Mbps 带宽不算宽裕，但只要配置得当，完全可以满足日常远程办公的需求。
